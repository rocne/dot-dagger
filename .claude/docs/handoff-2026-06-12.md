### #110 mechanics (most likely to matter later)

- `appConfig` gained provenance fields set in `resolvePaths`:
  `filesFromCwd` (true iff no `-f`, no `$DOTD_FILES`, no `$DOTFILES`, no
  config.yaml `dotfiles`) and `configExists`.
- `guardWalkSource(cfg)` returns a `hintError` when `filesFromCwd &&
  !configExists`; called at the top of `runPipeline`, `walkOrdered`, and
  `warnIfNosyncUnignored` (the last so the nosync pre-scan doesn't walk the
  cwd before the guard fires). Fresh machine now gets:
  `error: no dotfiles repo configured (refusing to walk the current directory)`
  + `hint: run 'dotd setup', or pass -f <path> / set $DOTFILES`.
- Apply warns when `activeCount > 0` but Links+Sourced+Generated == 0:
  `N active node(s) produced no actions — convention dirs may be missing
  .dagger files; run 'dotd init'` (closes the silent-no-op trap).
- `check: issues found` carries a `dotd setup` hint when config.yaml absent.
- Empty `list` / `env show` log an orienting note to **stderr** via
  `cfg.log.Infof` (stdout stays empty for pipes; `--json` unchanged; the
  `env show` note distinguishes file-missing vs file-empty).

  "first-run guardrails (2026-06-12 audit, PR 3)" banner.

## What was NOT done (deliberate)

- **P5** (only open audit finding): `unapply --all` (remove all symlinks)
  shadows the root persistent `--all` (show internal commands). Cosmetic;
  a rename is a breaking flag change — deferred.
- **Plan step 2b** (TTY-gating init's free-text name prompt): dropped on
  purpose. Integration tests (`TestInitAfterSetup`) and the documented
  driving pattern feed prompts via piped stdin lines (`y\n\ny\n…`);
  gating would break them. `-n` is the supported scripting path. The
  `yes | dotd init` → directory named `y` trap therefore still exists in
  principle, but is documented away by `-n`.
- **The `dot-` prefix semantics themselves** were not changed (U4 was
  fixed as documentation). `config/dot-tmux.conf` still maps to
  `~/.config/.tmux.conf`. If users trip on this despite the docs, a
  behavioral fix (e.g. top-level `~` mapping for `dot-` files) would be a
  spec-level change — brainstorm first.
- `maybeAddSourceLine` is a silent no-op when `shell` isn't in the
  resolved env (default env.yaml written by setup has only `os` +
  `hostname`). So `init -n` adds the RC line only when the user has
  `shell` configured. Pre-existing behavior, observed but not changed —
  arguably worth a future look (detect `$SHELL` at init time? — would
  touch the canonical-resolution rule, so design carefully).
