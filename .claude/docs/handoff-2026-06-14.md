## What this work is (the problem)

dot-dagger had **two unrelated things both named "link_root"**:
1. **Per-node `link_root:`** in each `.dagger` file — the legit, abstract linking
   mechanism (a dir declares where its files link). **Stays, untouched.**
2. **The global `link_root` knob** (`--link-root`/`DOTD_LINK_ROOT`/`config.yaml
   link_root`/setup prompt) → flowed into `expandDest` as `HomeDir`, i.e. it
   **redefined what `~` expands to**. The bug. User: "`~` must ALWAYS be `$HOME`.
   There is no link root."

That objection snowballed (correctly) into: **dot-dagger should put files where
the system already expects them and let the environment (XDG) decide — no
dot-dagger-specific path knobs at all.**

## The final design (what we're building)

**Three knob-less anchor tokens** in `.dagger` `link_root:`/dest values:

| Token | Resolves to | Configurable |
|-------|-------------|--------------|
| `~` | `$HOME` (`os.UserHomeDir`) | never |
| `$bin` | `($XDG_BIN_HOME ?: ~/.local/bin)/dot-dagger`, added to PATH | no (relocate via `$XDG_BIN_HOME`) |
| `$config` | `$XDG_CONFIG_HOME ?: ~/.config` | no (relocate via `$XDG_CONFIG_HOME`) |

**Tool-internal outputs**, also knob-less, under `$XDG_DATA_HOME`:
- generated staging → `$XDG_DATA_HOME/dot-dagger/generated`
- init.sh → `$XDG_DATA_HOME/dot-dagger/init.sh`

**Removed entirely** (flags + `DOTD_*` envs + config fields): `--link-root`,
`--bin-dir`, `--generated-dir`, `--init-file`, and the `link_root`/`bin_dir`/
`generated_dir` config.yaml fields. (`--config-dir` was considered and **rejected**
— see decision log.)

**Renamed** (tool's own state files — rare power-user, kept full flag+env
override): `--config`→`--dotd-config`, `--env-file`→`--dotd-env`. Envs stay
`DOTD_CONFIG_FILE`/`DOTD_ENV_FILE`.

**Added:**
- Anchor-token **validation** — unknown `~x`/`$x` tokens (typos like `$conifg`,
  or the old `~bin`) become a hard error instead of silently linking to a literal
  garbage path. Framed purely as typo-catching; **no migration language**.
- **`dotd paths`** — read-only resolved-view (home/`$bin`/`$config`/generated/
  init.sh/dotfiles/config.yaml/env.yaml), `--json`. Closes the introspection gap
  left by removing the config fields.

**config.yaml shrinks to a single `dotfiles` field.** **setup** shrinks to
prompting only for the dotfiles repo path (+ shell-RC source line). **No backward
compatibility** — stale `link_root:`/`bin_dir:` in config.yaml fails as a plain
unknown field (strict `KnownFields(true)`); no back-hinting. Tests configure the
**environment** (`t.Setenv("HOME"/"XDG_*")`, Docker `export`), never magic flags.

## Decision log (why, so it isn't relitigated)

- **`~` for home, `$bin`/`$config` for routes (not `~bin`/`~config`).** `~name` in
  real shells means "home of user name", so `~bin` masqueraded as that. `$`-tokens
  read as shell-variable placeholders, are literally env-backed, and `$` isn't
  special in YAML. Spelling is `config`, not `conf`.
- **Knob-less, pure-XDG (killed ALL path knobs).** User's reasoning: home is
  system-configured (`$HOME`), so is `$XDG_CONFIG_HOME`; a per-tool knob is
  redundant with the XDG environment. Verified: `unapply` removes links
  **per-link** (re-runs pipeline, `Readlink`s each dest), never via an owned dir —
  so the private bin dir bought zero cleanup safety; its only value was cosmetic.
- **`$bin` namespaced under `…/dot-dagger`, `$config` NOT.** PATH is a *search
  list* — a dedicated dir on PATH works identically while isolating dot-dagger's
  bins (namespacing is free). `$XDG_CONFIG_HOME` is a *direct lookup root* (apps
  read `~/.config/<app>`), so configs can't be namespaced. `$bin` namespacing =
  today's behavior, just knob-less + honors `$XDG_BIN_HOME`.
- **`--config-dir` knob rejected.** Was briefly designed (full knob like
  `bin_dir`), then dropped — redundant with `$XDG_CONFIG_HOME` and collided with
  `--config`. The earlier `--config`→`--dotd-config` rename now rests solely on
  the "mark as tool-internal / deprioritize" rationale (the collision is moot),
  which the user still wanted.
- **Verified facts:** `os.UserHomeDir()` returns `$HOME` verbatim on linux/darwin,
  errors only if unset, no `/etc/passwd` fallback (tested, Go 1.26.3).
  freedesktop XDG base spec defines NO bin dir; `~/.local/bin` is de-facto,
  `$XDG_BIN_HOME` an unofficial extension we honor.
