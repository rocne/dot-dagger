## Area: cmd/dotd

The `cmd/dotd` package is `package main` — the cobra-based CLI entry point for the `dotd` binary. All 19 `.go` files share the same package; there are no exported symbols (a `main` package), so "exported symbols" below lists the package-level identifiers that are visible across files in the package and the canonical contracts each file fulfils. Cross-file wiring is the dominant coupling pattern: `main.go` constructs the root command and every subcommand, and a small set of shared helpers (`resolveEnv`, `resolvePaths`, `runPipeline`, `filterWithPrompt`, `buildActOptions`, `annotateKeyError`, `promptConfirm`, `fileExists`, the `init`/`setup` prompt helpers) are reused across command files.

All paths below are relative to repo root `/home/rocne/git/dot-dagger`.

## Area: internal/config + internal/env + internal/ecosystem

These three packages form the **heart of the canonical-resolution design** described in
CLAUDE.md. `ecosystem` owns every `Default*()` path producer plus `ResolvePath`, the single
priority-chain resolver. `config` and `env` each wrap their YAML file (config.yaml / env.yaml)
and delegate path defaults back to `ecosystem`. The intended invariant: only
`resolvePaths()` in `cmd/dotd/main.go` (and a few documented bootstrap callers) may invoke
`Default*()` / `ResolvePath`; command code reads the already-resolved `cfg.*` values.

## Cross-cutting findings (overlap, Node model, mutations)

### 1. `.dagger` parsing is split across `dagger` + `pipeline`, NOT duplicated with `annotation`

There are effectively **two independent parsers** for `.dagger` files, but they parse different layers and do not overlap with `annotation`:

- **`dagger.Load`** parses the *YAML structure* (`gopkg.in/yaml.v3`, `dagger.go:64`). It stores `actions`, `when`, etc. as raw strings/lists — it does **not** interpret action syntax.
- **`annotation.Scan`** parses *file-header comments* (`# @key(args)`), an entirely different syntax (bufio line scanner, no YAML). No code path feeds one into the other.

The two reach a common representation only downstream in `pipeline`: `annotation`-derived keys are normalized to `Action` via `normalizeActionAnnotations` (`actions.go:13`) + `mergeActions`/`parseActionString` (`walk.go:411`), while `.dagger` `actions:` strings are parsed via `parseDaggerActions` (`walk.go:305`). The `BasicNode` YAML field set (`when/link_root/actions/after/require/request/disable`) is intentionally a 1:1 mirror of the annotation keys (`dagger.go:15` doc comment) — this is *parallel schema definition*, not parsing duplication.

### 2. Genuine duplicated parsing logic — the `key(args)` splitter exists THREE times

The "split `name(args)` on first `(` / last `)`" algorithm is implemented independently in three places with near-identical bodies:

- `annotation.parseKeyArgs` (`annotation.go:77-91`)
- `pipeline.parseActionString` (`walk.go:411-425`)
- `pipeline.parseDaggerActions`'s inline `link(...)` slice `s[5 : len(s)-1]` (`walk.go:317-319`) — a third, less-robust variant that only handles the `link(` prefix.

All three reduce to the same paren-extraction task; consolidating into one shared helper (e.g. in `annotation` or a small `syntax` util) would remove the divergence risk (the `parseDaggerActions` variant notably lacks the malformed-paren fallback the other two have).

### 3. Node model: there is no `node.Node` type — the real model is `pipeline.RawNode`

Despite the package name, `internal/node` exposes only `DeriveName`; the actual node struct is `pipeline.RawNode` (`walk.go:41-55`: `Path, LogicalName, EffectiveWhen, Actions, After, Require, Request, LinkRoot, LinkRootDir, IsCompose, ComposeTarget`). It is constructed in exactly two locations:

- `pipeline/walk.go` — three constructions: compose-target dir nodes (`:147`), regular file nodes (`:233`, after the 60-240 block), and `.dagger` `files:`-dict entries (around `:267`). All fields populated from cascaded `dirState` + annotations.
- `internal/adopter/adopter.go:109` — a *minimal* `RawNode` carrying only `Path`, `LogicalName`, `Actions` (no `EffectiveWhen`/`After`/`LinkRoot` etc.), built solely to drive `pipeline.Act` for the single adopted file.

`LogicalName` is always sourced from `node.DeriveName` unless overridden by `@name` (`walk.go:223`) or `.dagger` `name:` (`walk.go:145`).

### 4. Adopter filesystem mutations are real and ordered with a rollback path

`adopter.Adopt` is the only mutating member of these four packages. Order (`adopter.go:95-145`): `Stat` guard → (dry-run: `Act` only, no writes) → `copyFile` (`MkdirAll` + `O_EXCL` create preserving src mode + `io.Copy`, with `os.Remove(dst)` cleanup on copy error, `:168-195`) → `os.Remove(src)` → `pipeline.Act` (creates the symlink/source). If `Act` fails after the original is removed, it attempts `os.Rename(destAbs, src)` to restore, else emits a manual-recovery error (`:139-142`). Symlink creation itself is delegated entirely to `pipeline.Act` — adopter never calls `os.Symlink` directly.
