## Task 1: `dotd init` — Interactive Wizard

Prompts the user for:
1. Dotfiles repo path (default: `~/dotfiles`)
2. Bin dir (default: `~/bin`)
3. Config dir (default: `~/.config`)
4. Context values (comma-separated, e.g. `work,personal`)

Then:
- Writes `~/.config/dot-dagger/config.yaml` with provided values
- Writes `~/.config/dot-dagger/env.yaml` pre-populated with:
  ```yaml
  os: $(dotd get-os)
  hostname: $(dotd get-hostname)
  context: # set this to one of: work,personal
  ```
- If dotfiles repo exists: scaffolds `.dagger` files in detected or user-specified convention dirs
- Prints summary of what was created

### Prompting

Use simple `fmt.Print` + `bufio.NewReader(os.Stdin).ReadString('\n')` — no TUI library. Each prompt shows the default in brackets: `Dotfiles path [~/dotfiles]: `.

If the user presses enter with no input, use the default.

### Scaffolded `.dagger` files

`dotd init` does not create convention dirs. It writes `.dagger` files into directories that the user identifies as serving a convention role.

For each convention role the user wants to configure, prompt for the directory path within their dotfiles repo. Then write the appropriate `.dagger`:

**shellrc-role** `.dagger`:
```yaml
defaults:
  actions:
    - source
```

**conf-role** `.dagger`:
```yaml
defaults:
  actions:
    - link
```

**bin-role** `.dagger`:
```yaml
defaults:
  actions:
    - link
  link_root: ~/bin
```

Implement the prompt loop. Keep it linear: prompt → read → validate → next prompt. No backtracking needed.

Key functions:

## Task 2: `dotd list`

Runs pipeline through the order stage (env → walk → filter → order), then prints each active node.

Default output: one node per line, format `<logical-name>  <actions>  <path>`.

Flags:
- `--active` — show only active nodes (default if no filter flag)
- `--inactive` — show nodes that were filtered out by predicates (requires running filter with both outcomes)
- `--json` — output JSON array of node objects

- [ ] **Step 2.1: Create `cmd/dotd/list_cmd.go`**

```go
var listCmd = &cobra.Command{
    Use:   "list",
    Short: "List nodes and their status",
    RunE:  runList,
}

func init() {
    listCmd.Flags().Bool("active", false, "show only active nodes (default)")
    listCmd.Flags().Bool("inactive", false, "show nodes filtered out by predicates")
    listCmd.Flags().Bool("json", false, "output JSON")
}
```

For `--inactive`: run walk + evaluate each node's predicate, collect both active and inactive, tag them.

For `--json`: print a JSON array with fields: `logical_name`, `path`, `actions`, `active`, `when`.

- [ ] **Step 2.2: Register and smoke test**


## Task 3: `dotd bundle`

`dotd bundle <file>` takes a specific file in the dotfiles repo, finds all its DAG dependencies via `@after` annotations, and inlines them all into a single portable shell script.

Output order: dependencies first (in DAG order), then the target file.

Flags:
- `--output <file>` — write to file instead of stdout
- `--include-env` — prepend resolved env as `export KEY=val` lines at top

### Algorithm

1. Run pipeline through order stage to build the full DAG
2. Find the node for the given file
3. Walk the `@after` graph backwards to collect all transitive dependencies
4. Sort dependencies in DAG order (they're already ordered from step 1 — preserve relative order)
5. Concatenate: preamble (optional env exports) + dependency contents + target file content

- [ ] **Step 3.1: Update `cmd/dotd/bundle.go`**

The existing `bundle.go` from v1 may have different logic. Update to:
- Use the new pipeline's order stage to get the DAG
- Use new annotation `Args` field (not `Value`) for `@after` references
- Support `--include-env` flag

## Task 4: `dotd help --all`

`dotd help --all` reveals hidden commands (`get-os`, `get-hostname`) in help output.

Cobra's default help command does not support this natively. Two approaches:

**A. Custom help function:** Replace the default help with a function that checks for `--all`, and if set, sets `Hidden: false` on all commands before rendering.

**B. Separate `dotd internals` command:** A non-hidden command that lists internal commands with descriptions.

Approach A is more standard (matches the UX of `--all`).

- [ ] **Step 4.1: Implement custom help in `main.go`**

```go
rootCmd.SetHelpFunc(func(cmd *cobra.Command, args []string) {
    showAll, _ := cmd.Flags().GetBool("all")
    if showAll {
        for _, sub := range cmd.Commands() {
            sub.Hidden = false
        }
    }
    cmd.Usage()
})
rootCmd.PersistentFlags().Bool("all", false, "show all commands including internal")
```

Note: `--all` must be parsed before the help function runs. Cobra parses persistent flags early enough.

- [ ] **Step 4.2: Verify**

```bash
go run ./cmd/dotd help            # get-os and get-hostname NOT shown
go run ./cmd/dotd help --all      # get-os and get-hostname shown
go run ./cmd/dotd --help --all    # same
```

- [ ] **Step 4.3: Commit**

